# Adoption Difficulties Journal

This journal records process and difficulty evidence from first adoption. It does not replace `_TODO-repo-init.md` decision state, and it does not replace `.template-sync/marker.yml` sync state.

Use `_TODO-repo-init.md` for maintainer decisions and unresolved policy questions. Use `.template-sync/marker.yml` for retained modules, local overrides, protected-file decisions, and reviewed template commit state. Use this journal for what made the adoption hard, risky, blocked, or surprising.

After context loss, interruption, or compaction, reread this journal, `_TODO-repo-init.md`, `.template-sync/marker.yml` when present, and the current repository state before continuing.

### 2026-07-04 PR #5 initial adoption - Dependabot security settings were a manual GitHub.com step outside dependabot.yml

- **What happened:** Commit `4ee940f` added a manual `_TODO-repo-init.md` checklist item telling the maintainer to enable Dependabot alerts and, optionally, Dependabot security updates in GitHub.com repository settings. The checklist now distinguishes those repository settings from `.github/dependabot.yml`: the committed YAML controls routine version-update pull requests for retained ecosystems, while vulnerability alerts and automatic security-fix pull requests are separate GitHub security features that are not fully configured by committing `dependabot.yml`.
- **Why it mattered:** The upstream template ships a Dependabot configuration file, which can make adoption look complete once the file is tailored. In practice, this public curriculum repository still retains dependency surfaces for Markdown tooling, GitHub Actions, and pre-commit hooks, so dependency-security visibility matters even though the project has no application runtime. The template did not make the boundary between in-repo Dependabot configuration and out-of-repo GitHub security settings obvious enough for a downstream maintainer coming in cold.
- **Impact or risk:** A maintainer could merge the adopted repository believing Dependabot security handling is fully enabled because `.github/dependabot.yml` exists, while GitHub vulnerability alerts or security-update pull requests remain disabled. That leaves a gap between the repository's apparent security posture and the actual GitHub.com settings.
- **Resolution or workaround:** Added an explicit manual setup step in `_TODO-repo-init.md` for repository Settings > Code security and analysis. The checklist explains that `.github/dependabot.yml` is only one part of the dependency story and that Dependabot alerts/security updates must be reviewed separately in the repository UI.
- **Follow-up:** Template adoption guidance should treat Dependabot as two surfaces: committed routine-update configuration and GitHub.com security settings. When retaining Dependabot for any ecosystem, the initialization checklist should call out both surfaces and should enumerate the actual retained ecosystems from `.github/dependabot.yml`.
- **Related files:** `_TODO-repo-init.md`, `.github/dependabot.yml`
